## 1. Token Definitions

### Where tokens live

All design tokens are defined as **CSS custom properties inside `app/globals.css`**, scoped to two `@theme` blocks (Tailwind v4 convention — there is **no `tailwind.config.ts` file**).

| Source | Purpose |
|---|---|
| `app/globals.css` → `@theme` (top) | **Brand tokens** — primary, pastels, ink, line, font-sans, radii. These auto-generate Tailwind utilities (`bg-primary`, `text-pastel-blue-ink`, `rounded-xl`, `font-sans`). |
| `app/globals.css` → `@theme inline` | **shadcn primitive tokens** — `--color-background`, `--color-foreground`, `--color-primary-foreground`, sidebar/chart variables. Each maps to a `:root` CSS var below. |
| `app/globals.css` → `:root { ... }` | shadcn semantic values (oklch colors) for light mode. |
| `app/globals.css` → `.dark { ... }` | dark-mode override of `:root`. **Dark mode is not activated yet** but tokens are present. |

### Format

Brand tokens follow Tailwind v4's `--color-<name>` convention so utility classes auto-generate. Example:

```css
@theme {
  --font-sans: var(--font-inter), var(--font-noto-thai), system-ui, sans-serif;

  /* Royal blue brand */
  --color-primary: #2563eb;
  --color-primary-dark: #1e40af;
  --color-primary-50: #eff6ff;
  --color-primary-100: #dbeafe;
  --color-primary-200: #bfdbfe;

  /* Pastel accents (paired bg + ink for accessible contrast) */
  --color-pastel-blue: #dbeafe;       --color-pastel-blue-ink: #2563eb;
  --color-pastel-green: #dcfce7;      --color-pastel-green-ink: #16a34a;
  --color-pastel-amber: #fef3c7;      --color-pastel-amber-ink: #d97706;
  --color-pastel-pink: #fce7f3;       --color-pastel-pink-ink: #db2777;

  /* Neutrals */
  --color-ink: #0f172a;               /* primary text */
  --color-ink-muted: #64748b;         /* secondary text */
  --color-line: #e2e8f0;              /* borders, dividers */

  --radius-xl: 0.875rem;
  --radius-2xl: 1rem;
}
```

### Token-to-utility mapping (used across the codebase)

| Token | Generated utilities |
|---|---|
| `--color-primary` | `bg-primary`, `text-primary`, `border-primary`, `ring-primary` |
| `--color-primary-50` | `bg-primary-50`, `text-primary-50` |
| `--color-pastel-blue` / `-ink` | `bg-pastel-blue`, `text-pastel-blue-ink` (always paired) |
| `--color-ink` / `-muted` | `text-ink`, `text-ink-muted` |
| `--color-line` | `border-line`, `divide-line` |
| `--radius-xl` | `rounded-xl` |
| `--radius-2xl` | `rounded-2xl` |

### Transformation pipeline

**None.** Tokens are written directly in CSS and consumed by Tailwind v4's JIT engine at build time. No Style Dictionary, no transformation step.

### Spacing & sizing

We rely on **Tailwind's default spacing scale** (`p-4`, `gap-6`, `mb-20`, etc.) — no custom spacing tokens. Common values seen in this codebase:

- Page sections: `mb-20` (5rem) between major sections
- Card padding: `p-5` or `p-6`
- Container: `max-w-7xl mx-auto px-8`
- Gap inside grids: `gap-5` (cards) or `gap-3` (inline buttons)

### Typography scale

Body / utility text uses Tailwind sizes directly. Headings follow this convention:

| Element | Class | Token |
|---|---|---|
| Page title (h1) | `text-3xl font-extrabold tracking-tight` | — |
| Section title (h2) | `text-2xl` to `text-3xl` `font-extrabold tracking-tight` | — |
| Subsection (h3) | `text-base font-bold` or `text-xl font-bold` | — |
| Body | `text-sm` (default) or `text-base` for hero copy | — |
| Caption / meta | `text-xs text-ink-muted` | — |
| Eyebrow / label | `text-xs uppercase tracking-widest font-semibold text-ink-muted` | — |

## 2. Component Library

### Architecture (3 layers)

```
components/
├── ui/        ← Layer 1: shadcn/ui primitives (Radix-based, unstyled hooks)
├── layout/    ← Layer 2: site-wide chrome (Nav, Footer, LangSwitcher, StickyLine)
├── home/      ← Layer 3a: home-page sections (composes Layer 1 primitives)
├── tour/      ← Layer 3b: tour-related components (cards, counters, itinerary)
├── trip/      ← (reserved — not yet populated)
├── booking/   ← Layer 3c: booking checkout flow (step components + state hook)
└── car-rental/ ← Layer 3d: car rental form
```

**Rule:** Higher layers may import from lower layers, never the reverse.

### Layer 1 — `components/ui/` (shadcn primitives)

Generated by `npx shadcn@latest add <name>`. **Do not hand-edit unless absolutely necessary.** When you need a custom variant, wrap the primitive in a Layer-3 component instead.

Currently installed (15):
`accordion`, `badge`, `button`, `card`, `checkbox`, `dialog`, `dropdown-menu`, `form`, `input`, `label`, `radio-group`, `select`, `separator`, `sheet`, `tabs`

### Layer 2 — `components/layout/`

| Component | File | Type |
|---|---|---|
| `Nav` | `nav.tsx` | server component (uses `useTranslations` + `useLocale` from next-intl v4 — works in RSC) |
| `Footer` | `footer.tsx` | server component |
| `LangSwitcher` | `lang-switcher.tsx` | **client** (uses `useRouter`, `usePathname`, `onClick`) |
| `StickyLine` | `sticky-line.tsx` | **client** (window scroll listener, fixed position, global) |

### Layer 3 — feature components

#### `components/home/` (13 sections, in render order)

Hero · TrustStrip · QuickCategories · FeaturedTours · ToursByCountry · SchedulePreview · WhyGoTravel · Testimonials · PastToursGallery · TravelArticles · NewsletterStrip · Faq · FinalCta

Each is a **single-responsibility section** rendered once on the homepage. Reuse them inside other pages only if the design demands it.

#### `components/tour/`

`TourCard`, `TourCounter` (client, SWR-polled), `TourItinerary`, `TourPricingTiers`, `TourTripsList`, `ToursFilter` (client)

#### `components/booking/`

6-step checkout: `StepRail` (indicator), `StepSelection` → `StepTravelers` → `StepAddon` → `StepContact` → `StepReview` → `StepPayment`. State managed by `useBookingState` hook (client, sessionStorage-mirrored). Sidebar: `BookingSummary`. Shell: `CheckoutShell`.

### Component conventions

- **Default to server components** (no `'use client'` directive).
- **Mark `'use client'` only when needed**: for `useState`, `useEffect`, `onClick`, browser APIs, SWR.
- **Props are typed via the `type Props = { ... }` pattern** (inline at component top), not interfaces.
- **i18n-aware components** import from `next-intl`: `useTranslations(namespace)` + `useLocale()`.
- **Locale URL prefix** must be derived at component level: `const prefix = locale === 'en' ? '/en' : ''`.
- **Locale-dependent strings** prefer the i18n message catalog (`messages/{th,en}.json`); inline TH/EN ternaries are acceptable for small, one-off strings inside components.

### Documentation / storybook

**None.** Components are documented inline (TSDoc comments only when behavior is non-obvious). No Storybook. Keep this in mind when porting Figma designs — reference this doc for naming/structure.

## 4. Asset Management

### Images

**All tour images and gallery photos are remote URLs from Unsplash.** They are loaded through Next.js `<Image>` to get automatic AVIF/WebP, lazy loading, and responsive `srcset`.

`next.config.ts`:
```ts
images: {
  formats: ['image/avif', 'image/webp'],
  remotePatterns: [{ protocol: 'https', hostname: 'images.unsplash.com' }],
}
```

URLs are stored in:
- `data/tours.ts` → `coverImage`, `gallery[]`
- `data/articles.ts` → `cover`
- `data/past-tours-gallery.ts` → `image`
- `data/testimonials.ts` → `avatar`

**Local images** belong in `public/` (currently empty for tour content; favicon and shadcn assets only).

### Fonts

Loaded via **Next.js font system** in `app/[locale]/layout.tsx`:

```tsx
const inter = Inter({ subsets: ['latin'], variable: '--font-inter', display: 'swap' });
const notoThai = Noto_Sans_Thai({
  subsets: ['thai'],
  weight: ['400', '500', '600', '700', '800'],
  variable: '--font-noto-thai',
  display: 'swap',
});
```

The two CSS variables are exposed on `<html className={`${inter.variable} ${notoThai.variable}`}>` and consumed by `--font-sans` in `@theme`.

### CDN

None configured. Vercel's edge CDN serves all `_next/image`, `_next/static`, and remote `images.unsplash.com` proxied requests automatically once deployed.

### Optimization techniques

- Next.js `<Image fill sizes="..." className="object-cover" />` for all responsive imagery.
- AVIF/WebP automatic conversion at the optimizer.
- `priority` is **not** currently set on any image — consider adding it to the hero background for LCP.

## 5. Icon System

### Strict rule: **NEVER use emoji as icons.** Always use `lucide-react`.

This is a **non-negotiable project policy**. Documented in user memory, referenced in every component task. Rationale: emoji rendering is platform-dependent and inconsistent across OSes.

### Where icons live

`lucide-react` exports each icon as a named React component. Import inline at the top of any component file:

```tsx
import { Plane, Calendar, Users, MapPin, ArrowRight } from 'lucide-react';
```

### Sizing convention

| Context | Class |
|---|---|
| Inline with body text | `h-3 w-3` (12px) or `h-4 w-4` (16px) |
| Card icons / quick categories | `h-5 w-5` (20px) |
| Section heading / pillar icons | `h-6 w-6` (24px) or `h-7 w-7` |
| Decorative hero / large CTA | `h-8 w-8` and up |

### Icon-color pairing

Icons inherit `color: currentColor`, so set color via parent `text-*` utility:

```tsx
<div className="bg-pastel-blue text-pastel-blue-ink">
  <Plane className="h-5 w-5" />
</div>
```

### Frequently used icons in this project

`Plane`, `Calendar`, `Users`, `Car`, `MapPin`, `Clock`, `Star`, `Check`, `X`, `ArrowRight`, `ArrowUp`, `Globe`, `Search`, `Phone`, `Mail`, `MessageCircle`, `Quote`, `Shield`, `ShieldCheck`, `BadgeCheck`, `BadgeDollarSign`, `Award`, `QrCode`, `CreditCard`, `Building2`, `Minus`, `Plus`, `Download`

If you need an icon name not in `lucide-react`, reach for the closest semantic match — do **not** reach for emoji or another icon library.

## 6. Styling Approach

### Methodology: Tailwind utility classes (Tailwind v4 syntax)

- **No CSS Modules.** No styled-components. No CSS-in-JS.
- **No global styles** beyond `app/globals.css` (which is itself just `@theme` + a small `@layer base { ... }`).
- **Class composition via `cn()`** (`lib/utils.ts`, wraps `clsx` + `tailwind-merge`) — used by shadcn primitives.

### Responsive design

**Desktop-first design** but Tailwind utilities are mobile-first by default. Use breakpoint prefixes to layer up:

```tsx
className="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-4 gap-5"
```

Common breakpoints in use: `sm:` (640px), `md:` (768px), `lg:` (1024px). Container max width: `max-w-7xl` (1280px).

### Dark mode

**Not active.** shadcn's `.dark` selector is set up in `globals.css` but no toggle is wired. Brand tokens are light-only. Defer dark mode to Phase B/C.

### Animation conventions

- **Always use `transition-colors`, `transition-shadow`, or `transition-transform`** — never `transition-all`.
- **Duration:** 150–300ms. Use `duration-200` as default.
- **Hover effects** must NOT cause layout shift. Prefer color/opacity/shadow over `scale-*`.
- **`prefers-reduced-motion`** must be respected (Tailwind handles via `motion-safe:` / `motion-reduce:` variants when needed).

### A11y baseline

- All interactive elements use `cursor-pointer`.
- All clickable visuals (cards, gallery items) have visible hover state.
- Focus rings come from shadcn's `focus-visible:ring-3 focus-visible:ring-ring/50` defaults.
- Color contrast: text on `bg-white` uses `text-ink` (#0f172a, 15:1 contrast). Muted body text uses `text-ink-muted` (#64748b, 4.6:1).

### Feature organization rule

Code is grouped **by feature** under `components/<feature>/`, not by technical layer. A feature folder may contain its data hook, types, and UI together (e.g., `booking/use-booking-state.ts` lives next to its consumers).

### File naming

- Components: kebab-case `tour-card.tsx` exporting PascalCase `TourCard`.
- Hooks: kebab-case `use-booking-state.ts` exporting `useBookingState`.
- Types modules: kebab-case `booking-types.ts`.
- Data modules: kebab-case `past-tours-gallery.ts`.
- Tests: colocated, suffixed `*.test.ts` (vitest).

## How to use this doc when porting Figma designs

Workflow for the Figma MCP `figma:figma-implement-design` skill (or any agent):

1. **Read the Figma node** via `mcp__figma__get_design_context`.
2. **Map the design's tokens to ours** — the design's "primary" → `bg-primary` (royal blue `#2563eb`); design's "card surface" → `bg-white border border-line rounded-2xl`. If the Figma design uses different colors, prefer translating to nearest brand token rather than introducing new ones.
3. **Map components to existing primitives first** — does the Figma button match shadcn's `<Button>`? Does the input match `<Input>`? Reuse before creating new variants.
4. **Use lucide-react for every icon.** If the Figma design uses an icon set we don't have, find the nearest lucide equivalent.
5. **Place the new component in the correct layer** — site chrome → `components/layout/`; page section → `components/<page>/`; feature widget → `components/<feature>/`.
6. **Wire i18n** — never hardcode strings. Add to `messages/{th,en}.json` and use `useTranslations()`.
7. **Run `npm run build`** to verify the change compiles before committing.

## Non-negotiable rules (project policy)

1. **NO emoji ever.** All icons via `lucide-react`.
2. **No `tailwind.config.ts`.** Tailwind v4 — theme tokens live in `@theme` blocks in `app/globals.css`.
3. **next-intl v4 API only** — `getRequestConfig` uses `requestLocale: Promise<string>` (await it).
4. **`lib/store.ts` is the single migration point** — all booking/counter data goes through it. Phase B/C swaps in Prisma here.
5. **Server components by default** — only mark `'use client'` when there's a real reason (state, effects, browser APIs).
6. **No `transition-all`.** Use specific `transition-colors`, `transition-shadow`, `transition-transform`.
7. **Desktop-first responsive.** Use `md:`/`lg:` breakpoints to enhance, not `sm:` to shrink.
